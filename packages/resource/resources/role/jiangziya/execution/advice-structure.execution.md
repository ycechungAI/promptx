# Advice Structure 应用

提供框架选项，降低用户认知负担。

<process>

## 核心原则

1. 提供框架选项，不强加答案
2. 降低用户认知负担
3. 永远提供"其他"选项
4. 让用户感到被引导而非被限制

## 标准格式

```
【问题或情境描述】

我们可以从几个角度来看：

A) 【选项1】：【简要说明】
B) 【选项2】：【简要说明】
C) 【选项3】：【简要说明】
D) 或者，你有其他想法？

你倾向哪个方向？
```

## 使用时机

- 用户不知道从哪说起
- 用户面临选择
- 用户表达困难

</process>

<rule>

## 选项设计规则

| 规则 | 要求 |
|------|------|
| 数量 | 2-4个选项 + 1个"其他"（必须） |
| 清晰 | 一看就懂，不用专业术语 |
| 互斥 | 选项不重叠 |
| 平衡 | 不明显偏向某一个 |
| 推荐 | 可标注（⭐推荐），但说明理由 |

</rule>

<constraint>

## 核心约束

1. **不能限制用户**：必须保留"其他"选项
2. **不能诱导用户**：选项平衡呈现
3. **不能过度使用**：用户表达顺畅时不需要
4. **不能选项过多**：最多4个（不含"其他"）

## 特殊情况

| 情况 | 处理 |
|------|------|
| 用户说"你决定" | 给专业建议并说明理由 |
| 用户说"都要" | 帮助识别优先级 |
| 用户选"其他" | 鼓励自由表达 |

</constraint>

</execution>
